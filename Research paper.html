Abstract

Scam and phishing detection systems often rely either on rigid heuristic rules or opaque large language models. Heuristics lack generalization, while LLMs are costly and difficult to audit. This work presents a hybrid, interpretable scam detection pipeline that combines a feature-based supervised machine learning model, semantic analysis via an LLM safety model, and rule-based heuristics within a unified ensemble decision framework. The system achieves 94.64% accuracy, 94.62% F1, and 0.9812 ROC-AUC on a balanced benchmark of 1,400 messages, with a Brier Score of 0.049 confirming well-calibrated probability outputs.

§ 01

Introduction

Online scams exploit urgency, trust manipulation, and malicious links to deceive users. The FBI's Internet Crime Complaint Center reported over $12.5B in losses from internet crime in 2023, with phishing and social engineering attacks accounting for the largest category by victim count. These attacks exploit fundamental human cognitive biases: urgency, authority, and fear.

ApproachStrengthWeakness
Rule-based systemsPrecise, auditableBrittle; easily bypassed by novel attacks
ML classifiersGeneralizableOften opaque, lacking interpretability
LLM-based moderationSemantically powerfulExpensive, slow (~1s), inconsistent

This research explores whether a lightweight, interpretable ML model, combined with LLM-based semantic checks and heuristics, can provide robust scam detection without over-reliance on any single method.

1.1 Research Questions

(1) Can explicit feature engineering match or exceed deep learning performance on scam detection? (2) How do linear models compare to black-box approaches in terms of interpretability? (3) Can ensemble methods reduce single-point-of-failure risks in security systems?

1.2 Contributions

A 24-feature engineering framework capturing behavioral and structural scam signals · Interpretable logistic regression with coefficient-based explainability · Multi-signal ensemble strategy combining ML, heuristics, and LLM validation · Production-ready Flask microservice with sub-5ms inference latency
§ 02

Problem Formulation

Given an input text x, classify it into: Safe, Suspicious (requires human review), or Scam. The system is designed as a binary classifier with post-hoc three-tier bucketing based on calibrated confidence scores.

2.1 Formal Definition

Let x ∈ 𝒳 be a message and y ∈ {0, 1} be the binary label (0 = safe, 1 = scam). The system learns:

f : 𝒳 → [0, 1]    where f(x) = P(scam | x)

2.2 Optimization Objectives

(1) High recall for scam detection — minimize false negatives; (2) Low false positive rate — minimize user disruption; (3) Explainability — critical for security auditing and user trust.

Safety Priority: A false negative (missed scam) means a user loses money or has credentials stolen. A false positive is merely a minor inconvenience. The system is tuned to err on the side of caution.

Python base = GradientBoostingClassifier( n_estimators=200, max_depth=4, learning_rate=0.05, # conservative — avoids overfitting min_samples_leaf=4, subsample=0.8, # stochastic GB for variance reduction random_state=42 ) model = CalibratedClassifierCV( base, method="isotonic", cv=5 ) model.fit(X_train, y_train)

Raw GBM probabilities are often poorly calibrated. We apply isotonic regression — non-parametric, more flexible than Platt scaling — via 5-fold CalibratedClassifierCV. A score of 0.80 should correspond to ~80% actual scam rate.

The Brier Score of 0.049 confirms excellent calibration (perfect = 0; no-skill baseline for balanced classes = 0.25). The deployment API uses a threshold of 0.999 to achieve maximum precision for automated blocking.

Why isotonic over sigmoid? Sigmoid (Platt scaling) assumes a monotonic logistic transformation. Isotonic regression only assumes monotonicity — better suited to the step-function behavior of tree ensembles.

For the Logistic Regression interpretability companion:

P(y=1 | x) = σ(wᵀx + b)

Feature contribution for prediction i:

Contribution_i = w_i · x_i

This enables ranking features by influence on any individual prediction — analogous to SHAP for linear models, without the computational overhead.

Ensemble Decision Strategy

Final classification leverages multiple independent signals to reduce single points of failure. Heuristics provide fast initial filtering, the ML model offers interpretable predictions, and the LLM validates ambiguous cases.

8.2 Decision Logic

Python if ml_score > 0.90: verdict = "scam" # high confidence scam elif ml_score < 0.20 and heuristic_score < 30: verdict = "safe" # high confidence safe elif llm_unsafe or (ml_score > 0.70 and heuristic_score > 60): verdict = "scam" # ensemble agreement elif ml_score > 0.50 or heuristic_score > 50: verdict = "suspicious" # flag for human review else: verdict = "safe"
The ML model receives the highest weight (0.6) as it combines interpretability with strong generalization. This mirrors industrial fraud detection pipelines, emphasizing robustness over raw accuracy.
§ 09

Precision-Recall Trade-offs

The system uses three decision thresholds enabling flexible deployment tuning:

ZoneThresholdActionUse Case
Safe< 0.20Allow messageHigh-volume filtering
Suspicious0.20 – 0.90Flag for human reviewAmbiguous edge cases
Scam> 0.90Block / alert userAutomated blocking

The deployment API uses a conservative threshold of 0.999 to achieve maximum precision for automated blocking — trading ~2% recall for zero false positive auto-blocks. Security-sensitive deployments may lower the threshold to favor recall.

Calibration Quality: The Brier Score of 0.049 confirms well-calibrated probabilities. The 5-fold CV results (AUC: 0.9845 ± 0.0027) show consistent generalization with low variance — no severe overfitting.
§ 10

LLM-Based Semantic Safety Layer

Feature-based ML can miss: novel scam tactics not in training data, subtle persuasion and manipulation, and context-dependent deception. A Llama Guard model is integrated as a secondary semantic validator to capture these blind spots.

Design Principle: The LLM does not make final decisions — it acts as a semantic validator. Final authority remains with the interpretable ML model. This prevents over-reliance on expensive LLM calls and inconsistent LLM behavior.

§ 12

Limitations & Future Work

Current Limitations: English-only dataset · Dataset size (1,400) limits wider generalization claims · Static features miss temporal and contextual patterns · No image analysis for screenshot-based scams · Linear model restricts non-linear feature interaction capture

Expand dataset to 10K+ samples from real-world sources · Implement full SHAP value visualization · Add counterfactual explanations ("if you removed X, verdict would change") · Cross-validation with stratified k-fold splits

Gradient-boosted models (XGBoost, LightGBM) with native SHAP · Ensemble voting and stacking strategies · Domain reputation APIs (VirusTotal, URLhaus) · Multi-language support with language-specific feature sets

Online learning pipeline for continuous model updates · Graph-based features via email network analysis · Active learning with human-in-the-loop labeling · Adversarial robustness testing against deliberate obfuscation · Transfer learning from pre-trained security models

§ 13

Conclusion

This work demonstrates that interpretable machine learning, when integrated with LLM safety models and rule-based heuristics, can form a practical and auditable scam detection system. The system achieves 94.64% accuracy and 0.9812 ROC-AUC while maintaining full explainability via coefficient-based attribution.

Key Takeaways: Feature engineering outperforms black-box models for limited data regimes · Linear models provide sufficient performance with superior interpretability · Ensemble strategies offer robustness without over-reliance on any single component · Production-oriented design from day one enables real deployment

"I focused on interpretable ML for security. I designed explicit features, trained a linear classifier, analyzed feature contributions, and embedded the model into a multi-signal safety system rather than treating ML as a black box. The emphasis is on explainability, precision-recall trade-offs, and production-ready deployment — demonstrating end-to-end ML ownership, not just API usage."

— Vishwajeet Adkine, for MBZUAI Research Presentation
